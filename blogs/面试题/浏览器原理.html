# 浏览器安全

# 什么是XSS

代码注入攻击。攻击者通过在网站注入恶意脚本，使之在用户的浏览器上运行，从而盗取用户的信息如cookie等

本质是因为网站没有对恶意代码进行过滤，与正常的代码混合在一起了，浏览器没有办法分辨哪些脚本是可信的，从而导致了恶意代码的执行。

攻击者可以通过这种攻击方式进行以下操作

  • 获取页面的数据 DOM cookie localStorage
  • DOS攻击，发送合理请求，占用服务器资源，从而使用户无法访问服务器
  • 破坏页面结构
  • 流量劫持

经典表单 博客 之类 onerror 然后注入js代码 操作

防范

  • 对插入到HTML中的代码做好充分转意
  • 使用csp 本质是建立一个白名单，告诉浏览器哪些外部资源可以加载和执行，从而防止恶意代码的注入攻击

# 什么是CSRF

跨站请求伪造攻击

本质：利用 cookie 会在同源请求中携带发送给服务器的特点，以此来实现用户的冒充。

常见3种

  • get 比如在img标签构造一个请求，当用户打开这个网站就自动发起提交
  • post 比如构建一个表单，然后隐藏，当用户进入页面时，自动提交这个表单
  • 链接类型 比如在a的href构造一个请求，诱导用户点击

登录网站A 并在本地生成保存cookie 保持登录状态 此时访问b b发跨站请求A

防范

  • 进行同源检测
  • 使用CSRF token进行验证

# 进程与线程

# 进程与线程的概念

从本质上说，进程和线程都是CPU工作时间片的一个描述：

  • 进程描述了CPU在运行指令及加载和保存上下文所需的时间，放在应用上来说就代表了一个程序
  • 线程是进程中的更小单位，描述了执行一段指令所需的时间

进程是资源分配的最小单位，线程是cpu调度的最小单位

进程和线程之间的关系以下4个

  • 进程中的任意一线程执行出错，都会导致整个进程的崩溃
  • 线程之间共享进程中的数据
  • 当一个进程关闭之后，操作系统会回收进程所占用的内存
  • 进程之间的内容相互隔离

# chorm浏览器进程

  • 1个浏览器主进程
  • 1个GPU进程
  • 1个网络进程
  • 多个渲染进程
  • 多个插件进程

# 浏览器渲染线程有哪些

  • GUI渲染线程 负责渲染浏览器页面，解析HTML、CSS，构建DOM树、构建CSSOM树、构建渲染树和绘制页面，当界面需要 重绘 或由于操作引起 回流时，该线程就会执行

**注意：**GUI渲染线程和JS引擎线程是互斥的，当JS引擎执行时GUI线程会被挂起，GUI更新会被保存在一个队列中等到JS引擎空闲时立即被执行

  • JS引擎线程 也称JS内核 ，负责处理、解析js脚本，运行代码，js引擎一直等待着任务队列的到来
  • 时间触发线程 属于浏览器而不是JS引擎，用来控制时间循环
  • 定时器触发进程
  • 异步http请求进程

# 进程之间的通信方式

  • 管道通信 ：管道就是操作系统在内核中开辟的一段缓冲区，进程1可以将需要交互的数据拷贝到这段缓冲区，进程2就可以读取了
  • 消息队列通信
  • 信号量通信
  • 信号通信
  • 共享内存通信
  • 套接字通信

# 僵尸进程和孤儿进程是什么

  • 孤儿进程 父进程退出了，而他的一个或多个进程还在运行，那这些子进程都会称为孤儿进程，将北init进程收养，并由init进程对它们完成状态收集工作
  • 僵尸进程：子进程比父进程先结束，而父进程没释放子进程所占用的资源，那么子进程的进程描述符仍然在系统中，这种进程称之为僵死进程

# 死锁产生的原因？如何解决

死锁：多个进程在运行过程中因争夺资源而造成的一种僵局，当进程处于这种僵持状态时，若无外力作用，它们都将无法再向前推进。

资源分为两类

  • 可剥夺资源 指某进程再获得这类资源后，该资源可以再被其他进程或系统剥夺，CPU和主存均属于可剥夺资源
  • 不可剥夺资源，当系统把这类资源分配给某进程后，再不能强行收回，只能再进程用完后自行释放，如磁带机、打印机等

# 产生死锁的原因

  • 竞争资源
    • 竞争不可剥夺资源
    • 竞争临时资源
  • 进程间推进顺序非法

若P1保持了资源R1,P2保持了资源R2,系统处于不安全状态，因为这两个进程再向前推进，便发生死锁，当P1运行到P1:R2时，因R2已被P2占用而阻塞

# 产生死锁的必要条件

  • 互斥条件：进程要求对所分配的资源进行排他性控制，即在一段时间内某资源仅为一进程所用
  • 请求和保持条件：当进程因请求资源而阻塞时，对已获得的资源保持不放
  • 不剥夺条件：进程已获得的资源在未使用完之前，不能剥夺，只能在使用完时由自己释放
  • 环路等待条件：在发生死锁时，必然存在一个进程——资源的环形链‘

# 预防死锁的方法

记忆：破坏四个条件

  • 资源一次性分配：一次性分配素有资源，这样就不会再有请求了（破坏请求条件
  • 只要有一个资源得不到分配，也不给这个进程分配其他的资源（破坏保持条件
  • 可剥夺资源：当某进程获得了部分资源，得不到其他资源，则释放已获得资源
  • 资源有序分配法：系统给资源编号，进程按编号递增顺序请求资源，释放则相反

# 如何实现浏览器内多个标签页之间的通信

本质上是通过中介者模式来实现的，因为标签页之间没法直接通信，找中介者，让标签页和中介者进行通信，然后让这个中介者来进行消息转发

  • websocket协议

因为websocket协议可以实现服务器推送， 标签页向服务器发送数据，然后由服务器向其他标签页转发

  • 使用shareworker方式

shareworker会再页面存在的生命周期内创建一个唯一的线程，并且开启多个页面也只会使用同一格线程。这个时候共享线程就可以充当中介者的身份，标签页通过共享的这个线程实现数据交换

  • 使用local storage方式

再一个标签页对localstorage的变化事件进行监听，然后另一个标签页再修改数据时，可以通过监听事件来获取数据

  • 使用post message方法

如果我们能够获得对应标签页的引用，就可以使用postmessage办法，进行通信

# 浏览器缓存

# 对浏览器缓存机制的理解

  • 浏览器第一次加载资源，服务器返回200，浏览器从服务器下载资源文件，并缓存资源文件与response header , 以供下次加载时对比使用
  • 下一次加载资源时，由于强制缓存的优先级高。先比较当前时间与上一次返回200的 时间差，如果没有超过cache-control中设置的max-age，则没有过期 ， 命中强缓存，直接从本地读取资源，如果浏览器不支持HTTP1.1,则使用expires头判断是否过期
  • 如果资源已过期，则表明强缓存 没有命中，则开始 协商缓存，向服务器发送带有if-None-Match 和if-Modified-Sinced的请求
  • 服务器收到请求后，优先根据Etag值判断被请求的文件有没有做修改， Etag值一致说明没有修改，命中协商缓存，返回304，如果不一致则有改动，直接返回新的资源带上新Etag 返回200
  • 如果服务器收到的请求没有Etag值，则将If-Modified-Since和被请求文件的最后修改时间做对比，一致则命中协商缓存，返回304，不一致则返回新的last-modified和文件并返回200

很多网站的资源后面都加了版本号，这样做的目的是：每次升级了JS或CSS文件后，为了防止浏览器进行缓存，强制改变版本号，客户端浏览器会重新下载新的JS或CSS文件，以保证用户能够及时获得网站的最新更新

# 强缓存和协商缓存的区别

  • 强缓存

强缓存策略，如果有效，直接使用缓存资源，不必在向服务器发起请求。

两种方式来设置 http头信息中的 expires属性 和 cache-control属性

  • 服务器通过再响应头设置expires属性，来指定资源的过期时间。在过期时间以内，该资源可以被缓存使用，不必再向服务器发送请求，这个时间是一个绝对时间，它是服务器时间，因此可能存在问题，就是客户端的时间和服务端的时间不一致，或者用户对客户端时间进行修改，这样可能影响缓存命中的结果

  • expires是http1.0的方式，存在一些缺点，再http1.1中提出了一个新的头属性就是cache-control属性，他提供了对资源缓存更精确的控制，有以下属性

    • public ：设置了该字段值的资源表示可以被任何对象缓存，不常用，一般用max-age精确控制
    • private：设置了该属性的资源只能被用户浏览器缓存，不允许任何代理服务器缓存，在实际开发中，对于一些含有用户信息的HTML，通常都要设置这个字段值，避免CDN缓存
    • no-cache：设置该字段需要先和服务器确认返回的资源是否发生了变化，如果没变化，使用缓存好的资源（无强缓存，存在协商缓存
    • no-store：不缓存，每次都拉取最新的资源
    • max-age：设置缓存的最大有效期
    • s-maxage：优先级高于maxage和expires，仅适用于CDN

    一般来说只需要设置一种就可以实现强缓存，当两种方式一起使用，cache-control优先级高于expires

  • 协商缓存

使用协商缓存策略时，会先向服务器发送一个请求，如果资源没有改变，就返回304，让浏览器使用本地缓存，如果修改了，就返回修改后的资源

​ （1）服务器在响应头中添加last-modified属性，指出资源最后一次修改的时间，当浏览器下一次发起请求时，会在请求头添加if-modified-since ，值就是last-modified的值。当请求发送到服务器，服务i去会通过这个属性和最后一次修改时间对比，来判断是否更改，没更改返回304 ， 使用本地，更改返回新资源。 缺点是last-modified只能精确到秒级，会造成缓存命中的不准确

​ （2） 服务器返回资源时，在头信息添加Etag ，这个属性是资源生成的唯一标识，当资源改变，这个值也会改变。下一次请求时，浏览器会在请求头加一个if-none-match属性，值就是上次的etag， 服务器接收到后和资源当前的etag比较，来判断是否改变。比last-modified的方式更加精确

当last-modified和etag同时出现，etag优先级高

**总结：**两种缓存命中都会直接使用本地缓存，区别在于协商的回向服务器发送一次请求。他们缓存不命中时，都会向服务器发送请求来获取资源。在实际缓存机制中，两种策略是一起合作使用的。

  • 首先根据请求的信息判断是否命中强缓存，命中直接使用
  • 不命中则根据头信息向服务器发起请求，使用协商缓存
  • 如果协商命中，返回304 告诉浏览器直接使用本地
  • 不命中 返回最新资源

# 浏览器为什么需要缓存

  • 减少了服务器负担，提高了网站性能
  • 加快了客户端网页的加载速度
  • 减少了多余网络数据传输

# 点击刷新、F5、 CtrlF5 、 地址栏回车 有什么区别

  • 点击刷新或者f5 ：浏览器直接对本地的缓存文件过期，会带上If-Modified-Since，If-None-Match ，检查文件是否过期，返回可能是304 也可能是200
  • Ctrl F5强制刷新： 不仅会对本地文件过期，而且不会带上xxx，相当于之前没有请求过，返回结果是200
  • 地址栏回车：正常流程，走强缓存，协商缓存，然后返回
